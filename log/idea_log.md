# MediCare — 약 복용 알림 + 가족 모니터링

> 만성질환자가 약을 제때 챙겨 드시는지 가족이 함께 관리할 수 있는 헬스케어 케어 네트워크


> 가족 모니터링: 만성질환자(부모님 등)가 약을 제때 드셨는지 자녀가 실시간으로 확인하고, 미복용 시 자녀에게 알림을 보냅니다.
스마트한 알림 & 검색: 복잡한 약 복용 일정을 관리해 주고, 약의 부작용 및 상호작용 정보를 한눈에 제공합니다.
6개 마이크로서비스(MSA): 인증, 약 정보, 일정, 가족 권한 등 6개의 독립된 서비스로 나누어 설계하여 안정성과 확장성을 극대화했습니다.

## 🎯 우리가 해결한 문제

만성질환자(부모님 등)가 매일 정해진 시간에 여러 약을 복용해야 하지만,
- 자녀는 매번 전화로 확인해야 하고
- 본인은 잊기 쉽고
- 약 정보(부작용·상호작용)는 분산되어 찾기 어렵습니다.

이를 해결하기 위해 우리는 **6개의 마이크로서비스**로 구성된 시스템을 설계했습니다.

### ADR-1: 왜 마이크로서비스인가?
- **배경**: 팀원 6명이 각자 한 영역을 책임지고 싶었음
- **고려한 대안**: 단일 모놀리식, 모듈러 모놀리스
- **결정**: 마이크로서비스

### ADR-2: 왜 PostgreSQL + MongoDB + Redis 혼합인가?
- **배경**: 데이터 종류가 정형/비정형/실시간으로 다양
- **결정**: 3종 DBMS 혼합 사용
- **이유**: 
  - 정형 데이터(가족·복용기록) → PostgreSQL ACID
  - 비정형(약 정보) → MongoDB 유연 스키마
  - 알림 큐 → Redis 인메모리 속도

(결정 3~7개 동일 형식)

# WaitQ — 실시간 원격 웨이팅(줄서기) 플랫폼

맛집·팝업 앞에서 직접 줄 안 서고, 앱으로 원격 대기를 걸어 실시간으로 순번을 받는 서비스.

**핵심**
- 원격 대기 + 실시간 순번 알림
- 입장·취소·노쇼가 몰려도 순번이 안 꼬이게 처리 (Redis Sorted Set + 분산 락)
- 8개 마이크로서비스로 분리

**해결하는 문제**
맛집 줄서기로 시간 버리고, 종이 번호표는 노쇼·호출 누락 잦고, 오픈런 때 등록 몰리면 순번이 꼬임. → 원격으로 줄 서고, 동시 요청에도 순번이 정확히 유지되게.

**기술 스택**
- Redis: 대기열을 Sorted Set으로 두면 순번 자동 정렬 + 분산 락으로 동시성 처리. 위치 검색(Geo)·랭킹에도 활용
- GraphQL: 매장 화면(매장·메뉴·대기현황·리뷰)을 필요한 필드만 한 번에 조회
- Coroutine: 동시 등록·취소 비동기 처리
- DB 3종: PostgreSQL(회원·예약) / MongoDB(리뷰·알림 로그) / Redis(대기열·캐시)
- Spring Boot + Docker + MSA, 통신은 REST / GraphQL / WebSocket

**서비스 구성 (8개)**
User(인증) · Store(매장조회·GraphQL) · Queue(대기·순번) · Location(주변검색·Geo) · Notification(알림·WebSocket) · Owner(호출·입장·노쇼) · Review(리뷰) · Ranking(실시간 랭킹)

**포인트**
동시에 등록·취소·노쇼가 몰릴 때 순번이 꼬이는 race condition을 Redis Sorted Set + 분산 락으로 막음. 이걸 TDD(동시 100명 등록 → 순번 안 겹침)로 검증. 발표·트러블슈팅의 메인 소재.

# 💳 PayMap — 다계좌 구독·소비 통합 관리 앱

> 여러 통장과 카드에 흩어진 자동이체·구독료·소비 패턴을 한눈에 파악할 수 있는 개인 금융 관리 앱

---

## 어떤 서비스를 할 것인가

### 어떤 서비스가 존재하는가
- 토스, 카카오페이 — 연동 계좌 잔액·송금 중심, 소비 패턴 분석 기능 미흡
- 뱅크샐러드 — 자산 관리 중심이나 행동 기반 소비 패턴 분석은 부족
- **PayMap** — 여러 통장/카드에 분산된 모든 소비를 한데 모아, 정기결제·구독료 감지뿐 아니라 일상적인 소비 습관까지 분석해 주는 서비스. 예를 들어 "3일에 한 번꼴로 다이소에서 소비" 같은 행동 패턴, "월급날 다음날 지출 집중" 같은 날짜 기반 패턴 등 기존 앱들이 잘 안 해주는 "생활 밀착형 패턴 인사이트"에 집중

### 나는 어떤 불편함을 겪은 적 있나
- 통장을 여러 개 쓰다 보니 어느 통장에서 뭐가 빠져나가는지 한 번에 파악이 안 됨
- 구독료가 언제, 어느 통장에서 나가는지 기억이 안 나 잔액 부족이 생기는 경우가 있음
- 내가 몇 달째 쓰지도 않는 서비스에 돈이 빠져나가고 있어도 모르고 있는 경우가 많음
- 월급 들어오면 어디에 얼마를 쓰는지 패턴을 알고 싶지만 앱 여러 개를 왔다 갔다 해야 함

### 내가 정말 스마트폰에서 원하는 걸 할 수 있나
- 현재는 불가능. 통장마다 각 은행 앱을 따로 열어야 하고, 자동이체 항목을 한눈에 비교할 수 있는 앱이 없음
- PayMap을 쓰면 앱 하나로 모든 계좌의 정기결제·소비 흐름을 파악 가능

## 해당 서비스를 위해서 어떤 일을 해야하는가

### 어떤 요청이 들어오는가
- `GET /accounts` — 연동된 계좌/카드 목록 조회
- `GET /subscriptions` — 정기결제·자동이체 목록 조회
- `GET /transactions` — 거래내역 조회 (기간, 계좌 필터)
- `GET /insights/pattern` — 소비 패턴 분석 결과 조회
- `GET /alerts` — 앱 상단 알림 목록 조회 (출금 예정, 이상 소비 등)
- `POST /accounts/link` — 오픈뱅킹 계좌 연동

### 어떤 정보가 같이 들어올 수 있는가
- 계좌 조회 시: 사용자 ID, 조회할 계좌 ID
- 거래내역 조회 시: 시작일~종료일, 카테고리 필터, 계좌 ID
- 패턴 분석 시: 분석 기간 (최근 3개월, 6개월 등)
- 계좌 연동 시: 오픈뱅킹 인증 토큰

### 어떤 정보를 찾아야하는가
- 각 계좌의 거래내역 (금액, 날짜, 가맹점명)
- 반복 패턴이 있는 거래 (같은 가맹점, 비슷한 날짜, 비슷한 금액)
- 월별·주별 소비 흐름
- 특정 날짜 전후 소비 집중 여부 (예: 월급일 다음날)

### 어떤 데이터를 어디에서 찾아야하는가
- **오픈뱅킹 API (금융결제원)** — 사용자 동의 하에 여러 은행 계좌 거래내역 수집
- 수집된 데이터는 자체 DB에 저장 후 분석에 활용

### 찾은 데이터는 어떻게 가공해야하는가
- 거래내역에서 반복 패턴 감지: 같은 가맹점 + 날짜 편차 ±3일 이내 + 금액 유사 → 정기결제로 분류
- 카테고리 자동 분류 (식비, 교통, 구독, 쇼핑 등) + 사용자가 수정 가능
- 소비 패턴 분석: 며칠에 한 번 꼴로 얼마 이상 소비, 특정 날짜 전후 소비 집중도 계산
- 월별 동일 카테고리 증감률 계산

### 어떤 응답을 해야하는가
- 계좌별·카드별 자동이체 목록 (가맹점명, 금액, 출금 예정일, 출금 계좌)
- 소비 패턴 분석 결과 (텍스트 인사이트 + 차트 데이터)
- 앱 상단 알림 (출금 예정, 이상 소비 감지)
- 가계부 (기간별 수입·지출 합계, 카테고리별 분류)

### 어떤 정보를 응답에 포함시켜야만 하는가
- 출금 예정일, 출금 계좌, 가맹점명, 금액
- 패턴 분석 인사이트 (예: "5일에 한 번 꼴로 4만원 이상 소비", "월급일 다음날 소비 집중")
- 카테고리별 소비 비율

### 어떤 정보를 응답에 포함시키지 말아야만 하는가
- 계좌 비밀번호, 오픈뱅킹 인증 토큰 등 보안 정보
- 다른 사용자의 거래 정보
- 원시 거래내역 전체 (필요한 필드만 가공해서 반환)

## 마이크로서비스 구성

| 서비스 | 핵심 책임 | 사용 DB |
|--------|----------|--------|
| User | 회원가입·로그인·인증 | PostgreSQL |
| Account | 오픈뱅킹 연동·계좌 관리 | PostgreSQL |
| Transaction | 거래내역 수집·저장·조회 | PostgreSQL |
| Subscription | 정기결제 감지·분류 | PostgreSQL |
| Insight | 소비 패턴 분석 | Redis (캐시) |
| Notification | 앱 상단 알림 생성 | Redis |

---

## 기술 사용 이유

- **오픈뱅킹 API**: 여러 은행 계좌 거래내역을 사용자 동의 하에 한 번에 수집
- **Redis**: 패턴 분석 결과 캐싱 (매번 재계산 방지), 출금 예정 알림 스케줄링
- **GraphQL**: 계좌·거래·구독·분석 데이터를 화면에 필요한 필드만 한 번에 조회
- **TDD 적용 포인트**:
  - 정기결제 감지 로직 (날짜 편차, 금액 유사도 판단)
  - 소비 패턴 계산 로직 (며칠 간격, 특정 날짜 전후 집중도)
  - 카테고리 자동 분류 로직
