# MediCare — 약 복용 알림 + 가족 모니터링

> 만성질환자가 약을 제때 챙겨 드시는지 가족이 함께 관리할 수 있는 헬스케어 케어 네트워크


> 가족 모니터링: 만성질환자(부모님 등)가 약을 제때 드셨는지 자녀가 실시간으로 확인하고, 미복용 시 자녀에게 알림을 보냅니다.
스마트한 알림 & 검색: 복잡한 약 복용 일정을 관리해 주고, 약의 부작용 및 상호작용 정보를 한눈에 제공합니다.
6개 마이크로서비스(MSA): 인증, 약 정보, 일정, 가족 권한 등 6개의 독립된 서비스로 나누어 설계하여 안정성과 확장성을 극대화했습니다.

## 🎯 우리가 해결한 문제

만성질환자(부모님 등)가 매일 정해진 시간에 여러 약을 복용해야 하지만,
- 자녀는 매번 전화로 확인해야 하고
- 본인은 잊기 쉽고
- 약 정보(부작용·상호작용)는 분산되어 찾기 어렵습니다.

이를 해결하기 위해 우리는 **6개의 마이크로서비스**로 구성된 시스템을 설계했습니다.

## 💡 설계 결정 (ADR — Architecture Decision Records)

### ADR-1: 왜 마이크로서비스인가?
- **배경**: 팀원 6명이 각자 한 영역을 책임지고 싶었음
- **고려한 대안**: 단일 모놀리식, 모듈러 모놀리스
- **결정**: 마이크로서비스
- **이유**: 독립 배포·테스트·확장 + 팀 분담 명확

### ADR-2: 왜 PostgreSQL + MongoDB + Redis 혼합인가?
- **배경**: 데이터 종류가 정형/비정형/실시간으로 다양
- **결정**: 3종 DBMS 혼합 사용
- **이유**: 
  - 정형 데이터(가족·복용기록) → PostgreSQL ACID
  - 비정형(약 정보) → MongoDB 유연 스키마
  - 알림 큐 → Redis 인메모리 속도

(결정 3~7개 동일 형식)

# WaitQ — 실시간 원격 웨이팅(줄서기) 플랫폼

맛집·팝업 앞에서 직접 줄 안 서고, 앱으로 원격 대기를 걸어 실시간으로 순번을 받는 서비스.

**핵심**
- 원격 대기 + 실시간 순번 알림
- 입장·취소·노쇼가 몰려도 순번이 안 꼬이게 처리 (Redis Sorted Set + 분산 락)
- 8개 마이크로서비스로 분리

**해결하는 문제**
맛집 줄서기로 시간 버리고, 종이 번호표는 노쇼·호출 누락 잦고, 오픈런 때 등록 몰리면 순번이 꼬임. → 원격으로 줄 서고, 동시 요청에도 순번이 정확히 유지되게.

**기술 스택**
- Redis: 대기열을 Sorted Set으로 두면 순번 자동 정렬 + 분산 락으로 동시성 처리. 위치 검색(Geo)·랭킹에도 활용
- GraphQL: 매장 화면(매장·메뉴·대기현황·리뷰)을 필요한 필드만 한 번에 조회
- Coroutine: 동시 등록·취소 비동기 처리
- DB 3종: PostgreSQL(회원·예약) / MongoDB(리뷰·알림 로그) / Redis(대기열·캐시)
- Spring Boot + Docker + MSA, 통신은 REST / GraphQL / WebSocket

**서비스 구성 (8개)**
User(인증) · Store(매장조회·GraphQL) · Queue(대기·순번) · Location(주변검색·Geo) · Notification(알림·WebSocket) · Owner(호출·입장·노쇼) · Review(리뷰) · Ranking(실시간 랭킹)

**포인트**
동시에 등록·취소·노쇼가 몰릴 때 순번이 꼬이는 race condition을 Redis Sorted Set + 분산 락으로 막음. 이걸 TDD(동시 100명 등록 → 순번 안 겹침)로 검증. 발표·트러블슈팅의 메인 소재.

# 📚 [Project] BookBridge: 교내 전공 도서 및 강의 필기 공유 플랫폼

## 📌 1. 프로젝트 개요 (Project Overview)
*   **프로젝트명:** BookBridge (Campus Textbook & Lecture Note Sharing System)
*   **기획 배경:** 
    *   매 학기 비싼 전공책을 새로 구매하는 것은 학생들에게 큰 재정적 부담입니다. 또한, 같은 교내에서 선배들의 우수한 강의 필기나 요약본을 구하고 싶어 하는 수요가 많습니다.
    *   타 중고거래 앱과 달리, **'우리 대학교 학생 인증'** 및 **'당일 교내 직거래/대여'**에 특화되어 학기 초/시험 기간에 빠르게 전공 서적과 학습 자료를 공유할 수 있도록 돕습니다.
*   **설계 목적:** 본 프로젝트는 **Jira 기반의 철저한 이슈 관리**, **TDD(테스트 주도 개발)를 통한 핵심 도메인 로직 검증**, 그리고 코드 중심이 아닌 **시스템 아키텍처 및 설계 목적 명세**의 완성도를 극대화하는 데 초점을 맞춤니다.

본 시스템은 Native Android(Java/Kotlin) 클라이언트 프론트엔드와 도서 데이터 및 대여 상태를 관리하는 백엔드 서버로 구성됩니다.

### 🗂 사용 기술 명시 (Tech Stack)
*   **Frontend:** Android Studio (Java / Kotlin)
*   **Database & Test Library:** SQLite / Room DB (로컬 데이터 모델 검증), JUnit5, Mockito (단위 테스트 필수 라이브러리)
*   **Architecture Pattern:** 안정적인 상태 관리를 위한 MVVM 패턴 적용

### 💡 주요 단위 테스트 시나리오 (Unit Test Cases)

1.  **도서 대여 기간 및 연체료 계산 로직 테스트 (`RentalCalculatorTest`)**
    *   *Test 1 (정상 반납):* 반납 예정일 이내에 도서 반납 상태(`RETURNED`)로 변경 시, 연체료가 0원으로 정확히 계산되는지 검증.
    *   *Test 2 (연체 케이스):* 반납 예정일보다 3일 늦게 반납되었을 경우, 일일 연체료 규정에 따라 정확한 패널티 점수나 금액이 산출되는지 경계값 테스트.
2.  **게시글 유효성 검증 테스트 (`PostValidatorTest`)**
    *   *Test 1:* 전공책 등록 시 필수 항목(책 제목, 상태 사진, 가격 또는 대여료) 중 하나라도 누락되면 업로드 요청이 거절(`False`)되는지 단위 테스트 진행.
