## 1. 기획 문서

### 1-1. 문제 정의
- 맛집·팝업 앞에서 직접 줄 서면 시간 버리고 매장 앞도 붐빔
- 종이 번호표/구두 호출은 순번 분실·노쇼·호출 누락 잦음
- 오픈런처럼 짧은 시간에 등록 몰리면 순번 꼬임

### 1-2. 목표
- 앱으로 원격 대기 걸어서 현장 안 가도 순번 잡음
- 입장·취소·노쇼 생겨도 순번 자동 재정렬
- 순번 가까워지면 알림 받음

### 1-3. 사용자
- 손님: 대기 등록 / 순번 확인 / 알림 받기
- 매장 관리자: 호출 / 입장 처리 / 노쇼 처리

### 1-4. 핵심 기능 (MVP)
1. 회원가입 / 로그인
2. 주변 매장 검색(지도) + 매장 상세
3. 대기 등록 / 취소 / 내 순번 조회
4. 순번 알림(임박·입장 호출)
5. 매장 관리자용 호출·입장·노쇼 처리
6. 매장 리뷰·평점
7. (확장) 인기 매장 실시간 대기 랭킹

### 1-5. 이 주제 고른 이유
줄서기는 결국 "먼저 온 순서대로 줄 세운 목록"인데, 이게 Redis의 Sorted Set(점수 순으로 자동 정렬되는 자료구조)이랑 구조가 똑같음. 그래서 "왜 Redis 썼냐"고 물어보면 캐시 용도로 억지로 끼워 넣은 게 아니라 "줄서기 자체가 Redis 자료구조랑 맞아서"라고 바로 답할 수 있음.

또 등록·취소·노쇼가 한꺼번에 몰리면 순번이 꼬이는데, 이걸 안 꼬이게 처리하는 부분이 7주차에 배운 동시성(스레드/코루틴)을 제대로 써먹는 지점임. 발표 때 보여줄 핵심도 여기임.

## 2. 기술별 사용 이유 및 구조 설계

### 2-1. 시스템 아키텍처 (MSA)
서비스를 기능별로 쪼개서(MSA) 각자 독립적으로 돌게 함. 손님 요청은 API Gateway가 받아서 알맞은 서비스로 넘기고, 서비스끼리는 REST로 통신함. 알림처럼 실시간이 필요한 건 WebSocket이랑 Redis pub/sub으로 따로 뺐음.

(다이어그램 첨부 예정 — AWS diagram tool)

| 서비스 | 핵심 책임 | 저장소 | 주 통신 |
|---|---|---|---|
| API Gateway | 라우팅·인증 필터·진입점 | – | REST |
| User/인증 | 가입·로그인·JWT | RDBMS | REST |
| 매장(Store) | 매장·메뉴 정보 조회 | RDBMS | GraphQL |
| 웨이팅(Queue) | 대기 등록·취소·순번 정합성 | Redis(Sorted Set) + RDBMS | REST |
| 위치(Location) | 주변 매장 검색·지도 | Redis(Geo) | GraphQL |
| 알림(Notify) | 순번 임박·입장 호출 | NoSQL + Redis pub/sub | WebSocket |
| 매장관리(Owner) | 호출·입장·노쇼 처리 | RDBMS | REST |
| 리뷰(Review) | 후기·평점 | NoSQL | GraphQL |
| 실시간 현황 | 인기 매장 대기 랭킹 | Redis(Sorted Set) | REST |

### 2-2. 사용 기술과 선택 이유

| 기술 | 선택 이유 | 적용 위치 |
|---|---|---|
| Kotlin + Spring Boot | 코루틴 기본 지원, 비동기 유리 | 전 서비스 |
| RDBMS | 트랜잭션·일관성 필요 | 회원·매장·예약기록 |
| Redis | 대기열(Sorted Set), 분산 락, Geo, 랭킹 | 웨이팅·위치·현황 |
| NoSQL(MongoDB) | 형태 일정치 않은 로그·문서 담기 좋음 | 알림 이력·리뷰 |
| REST | 등록/취소/입장 같은 상태 변경 | 웨이팅·관리 |
| GraphQL | 중첩 데이터 필요한 필드만 조회 | 매장·위치·리뷰 |
| Coroutine | 동시 등록·취소 비동기 처리 | 웨이팅 |
| WebSocket | 순번 실시간 푸시 | 알림 |
| Docker | 서비스별 컨테이너화·배포 | 전체 |

중요한 세 개만 더 풀어쓰면:
- Redis: 대기열을 Sorted Set으로 두면 등록 시각을 점수로 줘서 순번이 알아서 정렬됨. 분산 락은 여러 요청이 같은 줄을 동시에 못 건드리게 잠그는 용도. 메모리에 올려놔서 빨라 실시간 순번 처리에 맞음.
- GraphQL: 매장 화면은 매장 정보·메뉴·대기 현황·리뷰가 한 화면에 다 뜨는데, REST면 이걸 여러 번 따로 불러와야 함. GraphQL은 필요한 것만 한 번에 골라 받을 수 있어서 화면 데이터 불러오기에 유리.
- Coroutine: 등록·취소가 동시에 쏟아질 때 스레드를 무겁게 안 쓰고 가볍게 비동기로 처리할 수 있음.

### 2-3. 저장소 셋으로 나눈 이유
같은 데이터베이스라도 잘하는 게 달라서 용도별로 나눴음.
- RDBMS: 회원·예약 기록처럼 틀리면 안 되는 데이터. 트랜잭션으로 정확하게 관리.
- Redis: 대기열·순번·위치 검색·랭킹처럼 속도가 중요한 것. 메모리에 올려서 빠름.
- NoSQL: 알림 기록·리뷰처럼 양 많고 형식이 자유로운 것. 스키마 안 정해도 돼서 편함.

### 2-4. API 구성
- GraphQL: 매장 → 메뉴 → 대기 현황 → 리뷰처럼 줄줄이 엮인 데이터를 필요한 것만 한 번에 가져올 때. 조회용.
- REST: 등록·취소·호출·입장처럼 "뭔가를 바꾸는" 요청. 명령용.
- WebSocket: 순번 임박/입장 호출처럼 서버가 손님한테 바로 밀어줘야 하는 알림.
- (GraphQL 스키마 / REST 명세 / ERD는 작성 예정)

## 3. 문제 상황 및 해결 (트러블슈팅)

### 3-1. 대기 순번 동시성 문제

무슨 문제냐면:
- 한 매장에서 입장 처리 + 취소·노쇼가 같은 순간에 겹치면 순번이 중복되거나 빠짐
- 순번을 그냥 리스트나 숫자 카운터로 관리하면, 여러 요청이 동시에 같은 값을 고치다가 계산이 꼬임 (이게 race condition)

어떻게 해결하냐면:
- 대기열을 Redis Sorted Set으로 둠. 등록 시각을 점수로 주면 알아서 순서대로 정렬되고, Redis 명령(ZADD 등)은 한 번에 하나씩 처리(원자적)돼서 동시에 들어와도 안 꼬임.
- 취소·재정렬처럼 꼬일 위험이 있는 구간만 분산 락으로 잠가서 한 번에 한 요청만 들어가게 함.
- 처리는 코루틴으로 비동기로 돌려서 요청 많아도 버팀.
- 노쇼는 호출하고 일정 시간 안 오면 자동으로 다음 사람한테 넘김.

```text
// 대기 등록 — 중복 막고 등록 시각으로 순번 정렬
function enqueue(storeId, userId):
    key = "waiting:" + storeId
    if ZSCORE(key, userId) exists:
        return ALREADY_QUEUED
    ZADD(key, currentTimeMillis(), userId)
    return ZRANK(key, userId) + 1

// 취소 — 충돌 구간만 락으로 보호
function cancel(storeId, userId):
    lock = acquireDistributedLock("lock:waiting:" + storeId)
    try:
        ZREM("waiting:" + storeId, userId)   // 뒷순번은 자동으로 당겨짐
    finally:
        releaseLock(lock)
```

- 검증은 5-2 테스트 케이스로 함

### 3-2. 개발 중 발생 사례
- 기술 합치거나 협업하면서 실제 겪은 문제 여기 정리 (예: 서비스 간 통신 타임아웃, GraphQL N+1, Redis 락 데드락, Docker 네트워크 — 작성 예정)

## 5. TDD 적용

### 5-1. 테스트 크기 구분 (small / medium / large)
테스트가 뭘 건드리느냐로 셋으로 나눔. 작을수록 빠르고 자주 돌릴 수 있음.

| 크기 | 외부 의존 | 예시 (웨이팅) |
|---|---|---|
| small | 없음 (Mock·인메모리) | 순번 계산, 중복 판정 같은 순수 로직 |
| medium | 로컬 DB·Redis (Testcontainers) | 리포지토리·서비스 계층 |
| large | 여러 서비스 + 네트워크 (E2E) | 등록→호출→입장 흐름, 동시성 부하 |

- small: 외부에 의존 안 하고 로직만 보는 거라 제일 빠름. 제일 많이 만듦.
- medium: DB나 Redis를 실제로 띄워서 같이 도는지 확인.
- large: 서비스 여러 개를 진짜로 연결해서 전체 흐름 확인. 느리지만 실제 동작 검증용.

### 5-2. 핵심 테스트 케이스

| 테스트 | 조건 | 기대 결과 |
|---|---|---|
| 순번 무결성 | 동시 100명 등록 | 순번 1~100 중복·누락 없음 |
| 취소 재정렬 | 중간 순번 동시 취소 | 뒷순번 자동 당김 |
| 중복 등록 | 같은 유저가 두 번 등록 | 한 건만 허용 |
| 노쇼 | 호출 후 미입장 | 자동으로 다음 순번 호출 |

- 기능별 단위 테스트는 구현하면서 테스트 먼저(Red→Green) 작성함
