#### 협업 도구 활용
- git
	- 자기가 작성, 공부한거 정리해서 이 레포에 커밋해 넣어 기록 남겨가며 정보 공유하기
	- → 각자 자기 브랜치를 파서 거기서 작업함. 공부하거나 정리한 내용을 커밋으로 남기는데, 커밋 메시지 앞에 Jira 이슈 키(예: `KAN-12`)를 붙여서 어떤 이슈 작업인지 추적되게 함. 작업이 끝나면 PR을 올려 리뷰 받고 메인에 머지. 이렇게 하면 누가 언제 뭘 했는지 기록이 남아서 정보 공유가 됨.
- jira
	- 직접 epic 작성하기 (가능하면 본인이 고안한 `Micro Service`에 대해 잘 작성해보기)
	- → Epic은 큰 덩어리(예: "웨이팅 서비스"), 그 아래 Story/Task로 세부 작업을 쪼갬. 이슈마다 `[설계]`/`[기능]`/`[TDD]` 라벨을 붙여 종류를 구분함. 본인 마이크로서비스의 Epic을 직접 만들고, 거기에 들어갈 작업을 Story로 나눠 적으면 됨.
- AWS archetecture diagram
	- 만들어오기
	- → 우리 서비스 구조를 그림 한 장으로 보여주는 것. 서비스(박스)들, 각 서비스가 쓰는 DB, 서비스끼리 어떻게 통신하는지(화살표)를 그림. visual-paradigm 같은 무료 다이어그램 툴로 만들 수 있음.

#### 개발과정 참여 방법

> 개인적으로 각자 하나씩 Micro Service를 만들어 넣으면 팀 성적도 개인 보너스 점수도 같이 올라갈 수 있다고 생각합니다. 화이팅!

- 어떤 서비스를 할 것인가
	- 어떤 서비스가 존재하는가
		- → 테이블링·캐치테이블처럼 식당 원격 줄서기(웨이팅) 앱이 이미 있음. 우리는 이걸 백엔드 관점(MSA·동시성 처리)에서 다시 설계해보는 것.
	- 나는 어떤 불편함을 겪은 적 있나
		- → 맛집 앞에서 한참 줄 서거나, 종이 번호표 들고 기다리다 순서를 놓친 경험. 원격으로 줄 서고 순번 알림을 받으면 그 시간을 안 버려도 됨.
	- 내가 정말 스마트폰에서 원하는걸 할 수 있나
		- → 앱으로 대기를 걸고, 내 순번을 실시간으로 보고, 차례가 가까워지면 알림 받는 것. 충분히 스마트폰으로 할 수 있는 일임.

- 해당 서비스를 위해서 어떤 일을 해야하는가  *(웨이팅 서비스 기준 예시)*
	- 어떤 요청이 들어오는가
		- → "이 매장 대기 걸어줘", "대기 취소해줘", "내 순번 알려줘" 같은 요청.
	- 어떤 정보가 같이 들어올 수 있는가
		- → 매장 ID, 사용자 ID, 대기 인원 수 등.
	- 어떤 정보를 찾아야하는가
		- → 그 매장 대기열에 이미 등록돼 있는지, 지금 내가 몇 번째인지.
	- 어떤 데이터를 어디에서 찾아야하는가
		- → 대기열·순번은 Redis(Sorted Set)에서, 매장 기본 정보는 RDBMS에서.
	- 찾은 데이터는 어떻게 가공해야하는가
		- → Redis에서 내 순위를 꺼내 1을 더해 "n번째"로 만들고, 필요하면 예상 대기 시간도 계산.
	- 어떤 응답을 해야하는가
		- → 등록 성공 여부 + 내 순번 (+ 예상 대기 시간).
	- 어떤 정보를 응답에 포함시켜야만 하는가
		- → 내 순번, 등록 상태.
	- 어떤 정보를 응답에 포함시키지 말아야만 하는가
		- → 다른 손님의 개인정보, 내부 DB 키 같은 건 빼야 함.

- API를 구현하기 위해선 어떻게 해야할까?
	- HTTP API란?
		- → 앱(클라이언트)과 서버가 HTTP 규약으로 데이터를 주고받는 통로. 앱이 요청을 보내면 서버가 응답을 돌려줌.
	- REST API란?
		- → HTTP를 쓰는 대표적인 방식. 데이터를 "자원"으로 보고 URL로 가리키며, GET(조회)/POST(생성)/PUT(수정)/DELETE(삭제) 메서드로 다룸. 보통 JSON으로 주고받음.
	- spring boot라는 framework에선 api를 어떻게 작성해야할까?
		- → `@RestController` 클래스를 만들고, `@GetMapping`/`@PostMapping` 같은 어노테이션으로 URL과 메서드를 연결함. 요청 본문은 `@RequestBody`, URL 경로 값은 `@PathVariable`로 받고, 객체를 return하면 자동으로 JSON으로 변환돼서 응답됨.

- TDD를 해보려고 노력한다면 어떻게 해야할까?
	- 어떤 Test를 만들어야할까?
		- → 핵심 로직이 의도대로 도는지 확인하는 테스트. 예: 동시에 100명이 등록해도 순번이 안 겹치는지, 중복 등록이 막히는지. 기능을 만들기 "전에" 테스트부터 작성함(실패=Red → 통과=Green).
	- small/medium/large는 어떤 기준으로 분리되는걸까?
		- → 테스트가 외부(네트워크·DB)에 얼마나 의존하느냐로 나눔.
		- → small: 외부 의존 없이(Mock) 로직만 봄. 제일 빠르고 많이 만듦.
		- → medium: 로컬 DB·Redis를 실제로 띄워서 같이 도는지 확인.
		- → large: 여러 서비스를 진짜로 연결해 전체 흐름을 확인. 느리지만 실제 동작 검증용.

- GraphQL
	- 뭘까?
		- → 클라이언트가 "필요한 데이터의 모양"을 직접 정해서 요청하는 API 방식. 엔드포인트 하나로 다양한 조회를 함.
	- 어케 쓸까?
		- → 받고 싶은 필드를 쿼리로 적어 보냄. 예: 매장 이름·대기 인원만 달라고 하면 딱 그것만 응답으로 옴.
	- 왜 쓸까?
		- → REST는 화면에 필요한 데이터를 여러 번 나눠 부르거나, 필요 없는 것까지 받게 됨(over-fetching). GraphQL은 필요한 것만 한 번에 받아서 효율적임.
	- 그래서 이건 우리 프로젝트에서 왜 필요할까? (어떤 역할을 할 까?)
		- → 매장 화면이 매장 정보·메뉴·실시간 대기 현황·리뷰를 한 번에 보여줘야 함. 이걸 GraphQL로 필요한 필드만 한 방에 가져오는 역할.

- 어떤 DBMS를 쓸까?
	- 어떤 DBMS가 존재하는가?
		- → 크게 세 종류. RDBMS(MySQL·PostgreSQL), NoSQL(MongoDB 등), 인메모리(Redis).
	- 어떤 역할인가? (언제 사용하는가?)
		- → RDBMS: 회원·예약처럼 틀리면 안 되는 데이터. 트랜잭션으로 정확하게 관리.
		- → Redis: 대기열·순번·랭킹·위치 검색처럼 빠른 처리가 중요한 것. 메모리에 올려서 빠름.
		- → NoSQL: 알림 기록·리뷰처럼 양이 많고 형식이 자유로운 것.

##### Server
- container, container ochestration
	- → 컨테이너(Docker): 앱과 실행 환경을 한 묶음으로 포장해서 어디서든 똑같이 돌게 함. 오케스트레이션(Docker Compose·Kubernetes): 여러 컨테이너를 한꺼번에 띄우고 연결·관리해줌.
- lang / lib / framework
	- → Kotlin(또는 Java) + Spring Boot. 코루틴으로 비동기 처리.
- API
	- → REST(상태 변경 명령) + GraphQL(중첩 데이터 조회) + WebSocket(실시간 알림).
- TDD
	- → JUnit 등으로 테스트를 먼저 작성(Red→Green)하고, small/medium/large로 나눠 관리.
- DBMS
	- → RDBMS(핵심 데이터) + Redis(대기열·캐시·랭킹) + MongoDB(로그·리뷰).
