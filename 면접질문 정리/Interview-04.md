# Interview Question 04

## 주제

### print() 대신 Log를 사용해야 하는 이유

---

# 질문

프로젝트를 진행하다 보면 log를 확인하면서 개발을 해야만 합니다.

간혹 print() 등의 함수를 사용하여 바로 로그를 확인하거나,

log4j, SLF4J 등의 로그 라이브러리를 사용하여 로그를 출력합니다.

---

## print() 함수 대신 Log를 사용해야 하는 이유를 설명해주세요.

---

# 답변

프로그램 개발 과정에서 현재 상태를 확인하거나 문제를 추적하기 위해 로그(Log)를 사용한다.

간단한 개발 단계에서는 print() 함수를 사용할 수 있지만, 실제 서비스 환경에서는 전문적인 로그 라이브러리를 사용하는 것이 일반적이다.

대표적으로

- SLF4J
- Logback
- Log4j

등이 사용된다.

---

# 1. 성능 (Performance)

## print()의 문제점

print() 함수는 대부분 동기적(Synchronous)으로 동작한다.

즉,

```text
로그 출력

↓

콘솔 출력 완료

↓

다음 코드 실행
```

순서로 진행된다.

---

### 문제

콘솔 출력은 I/O 작업이다.

I/O 작업은 CPU 연산보다 상대적으로 느리다.

따라서 대량의 print() 호출은 애플리케이션 성능을 저하시킬 수 있다.

---

## 로그 라이브러리의 장점

대부분의 로그 라이브러리는 비동기 로깅(Async Logging)을 지원한다.

```text
로그 생성

↓

버퍼 저장

↓

백그라운드 스레드 출력
```

방식으로 동작한다.

---

### 결과

- 응답 속도 향상
- 서버 부하 감소
- 성능 저하 최소화

---

# 2. 유연성 (Flexibility)

로그 라이브러리는 다양한 설정을 지원한다.

---

## 로그 레벨

### TRACE

가장 상세한 로그

---

### DEBUG

개발 단계 디버깅 로그

---

### INFO

일반적인 정보 로그

---

### WARN

주의가 필요한 상황

---

### ERROR

오류 발생

---

## 예시

```kotlin
logger.info("회원가입 성공")

logger.warn("중복 이메일 요청")

logger.error("DB 연결 실패")
```

---

### 장점

운영 환경에서는

```text
DEBUG 제거

INFO 이상만 출력
```

가능

---

반면 print()는

```text
모든 로그 동일
```

하다.

---

# 3. 유지보수성 (Maintainability)

서비스 운영 과정에서는 문제가 발생한 원인을 추적해야 한다.

---

## 로그 활용

```text
사용자 로그인

↓

DB 조회

↓

Redis 조회

↓

JWT 발급

↓

응답
```

---

과정 전체를 추적할 수 있다.

---

### 예시

```text
[INFO]
User Login Start

[INFO]
Redis Session Found

[INFO]
JWT Generated
```

---

### 장점

문제 발생 시

```text
언제

어디서

무슨 이유로
```

오류가 발생했는지 확인 가능

---

## print()의 한계

운영 서버에서는 콘솔 출력이 보존되지 않거나 관리하기 어렵다.

---

# 4. 보안 (Security)

로그 라이브러리는 민감한 정보를 관리할 수 있다.

---

## 예시

### 잘못된 방식

```text
password=1234
```

---

### 올바른 방식

```text
password=****
```

---

### 장점

- 개인정보 보호
- 비밀번호 보호
- 보안 정책 준수

---

# 로그 라이브러리 종류

## SLF4J

Simple Logging Facade for Java

---

### 특징

- 추상화 계층 제공
- Logback, Log4j 교체 가능
- Spring Boot 기본 사용

---

## Logback

Spring Boot 기본 로그 구현체

---

### 특징

- 성능 우수
- 설정 편리
- XML 설정 지원

---

## Log4j2

고성능 로그 라이브러리

---

### 특징

- Async Logging 지원
- 대규모 서비스 적합

---

# 발표 예상 질문

## Q1. print() 대신 Log를 사용하는 이유는 무엇인가요?

### 답변

로그 레벨 관리, 성능 최적화, 유지보수성, 보안 측면에서 훨씬 우수하기 때문입니다.

---

## Q2. Spring Boot에서 기본적으로 사용하는 로그 라이브러리는 무엇인가요?

### 답변

SLF4J와 Logback 조합을 기본적으로 사용합니다.

---

## Q3. 로그 레벨에는 어떤 종류가 있나요?

### 답변

TRACE, DEBUG, INFO, WARN, ERROR 등이 있습니다.

---

## Q4. User Service에서 로그는 어디에 활용할 수 있나요?

### 답변

회원가입, 로그인, 인증, GraphQL 조회 과정의 흐름을 추적하고 장애 발생 시 원인을 분석하는 데 활용할 수 있습니다.

---

## Q5. 운영 환경에서 print()를 사용하면 안 되는 이유는 무엇인가요?

### 답변

성능 저하, 로그 관리 어려움, 로그 레벨 제어 불가, 보안 문제 등이 발생할 수 있기 때문입니다.

---

# 실무 관점

실제 기업에서는 print()를 거의 사용하지 않는다.

대신

- SLF4J
- Logback
- Log4j2

등을 사용하여 로그를 체계적으로 관리한다.

로그는 장애 분석, 성능 분석, 보안 감사, 사용자 행동 분석 등 다양한 목적으로 활용된다.

---

# 생활 비유

print()는 종이에 메모를 적어 책상 위에 올려두는 것과 같다.

당장은 보기 쉽지만 시간이 지나면 관리가 어렵다.

반면 로그 시스템은 날짜별, 중요도별로 정리되는 CCTV 기록 시스템과 같다.

문제가 발생했을 때 언제, 어디서, 무엇이 발생했는지 추적할 수 있기 때문에 실제 서비스 운영에서는 로그 시스템이 필수적이다.

---

# 한 줄 요약

print()는 간단한 디버깅에는 유용하지만, 실제 서비스 환경에서는 성능, 유지보수성, 보안, 로그 관리 측면에서 SLF4J, Logback, Log4j와 같은 전문 로그 라이브러리를 사용하는 것이 바람직하다.
