## AWS 클라우드 인프라 아키텍처 명세 (MSA)

본 인프라는 가용성 보장, 트래픽 분산, 데이터 격리 보안을 목표로 설계되었으며, VPC 내부를 Public, Private(App), Private(DB)로 구획한 표준 3-Tier 구조를 따릅니다. 특히 동네 주민들이 동시에 접속하는 **"선착순 타임딜 공동구매"** 환경에서 대규모 트래픽을 안정적으로 제어하도록 구성되었습니다.

<hr>

### 1. 엔드 투 엔드 데이터 흐름 (Data Flow)

1. **인입:** 모바일 클라이언트 요청 ──▶ **Amazon Route 53** (DNS 매핑)

2. **분산 및 관문:** **AWS ALB** (부하 분산) ──▶ **AWS API Gateway** (최외곽 Public Subnet `10.0.1.0/24`)

3. **라우팅:** API Gateway ──▶ 사설망 내부 Private Subnet - App (`10.0.2.0/24`) 내 해당 마이크로서비스로 전달

4. **데이터 접근:** 애플리케이션 ──▶ Private Subnet - DB (`10.0.3.0/24`) 내 **Amazon ElastiCache** 및 **Amazon RDS** 참조

<hr>

### 2. 레이어별 핵심 역할 및 실무 작동 방식

#### A. Public 레이어 (인입 및 트래픽 제어 관문)

* **Amazon Route 53**
  * **인프라 역할:** 도메인 질의를 최적의 엔드포인트 IP로 라우팅합니다.
  * **서비스 작동:** 사용자가 앱을 켜고 접속할 때 우리 서비스의 가상 영토로 정확하게 진입할 수 있도록 나침반 역할을 수행합니다.
  
* **AWS ALB (Application Load Balancer)**
  * **인프라 역할:** L7 영역에서 트래픽을 분산하여 단일 장애점(SPOF)을 제거합니다.
  * **서비스 작동:** 타임딜 마감 직전 수천 명의 사용자가 동시에 접속할 때 발생하는 폭발적인 부하를 여러 차선으로 안전하게 나누어 감당합니다.
  
* **AWS API Gateway**
  * **인프라 역할:** 백엔드 마이크로서비스의 IP를 은닉하고, 통합 인증/인가 및 동적 라우팅을 수행하는 단일 진입점입니다.
  * **서비스 작동:** 요청을 가장 먼저 받아 인증을 검증한 뒤, 회원 관련 요청은 User로, 공구 주문 요청은 Order로 신속하게 분류하여 내부 사설망으로 넘겨줍니다.

<hr>
#### B. App 레이어 (핵심 비즈니스 로직 처리)
* **Private Subnet - App (`10.0.2.0/24`) 환경**
  * **인프라 역할:** 외부 인터넷과 차단된 사설 대역으로, `User`, `Order`, `Payment` 서비스가 개별 컨테이너로 구동되어 장애 확산을 방지하고 독립적인 수평 확장(Horizontal Scaling)을 가능하게 합니다.
  

* **User Service (최다울 담당)**
  * **서비스 작동:** 유저의 동네 GPS 인증이나 로그인 요청을 Coroutine 비동기 기술을 통해 지연 없이 실시간으로 처리합니다.
  
* **Order & Payment Service (조원 담당)**
  * **서비스 작동:** 선착순 상품 수량이 마감되기 전, 주문서 생성과 결제 승인 연산을 각각 독립된 프로세스에서 상호 간섭 없이 안전하게 처리합니다.

<hr>
#### C. DB 레이어 (데이터 초고속 처리 및 영속성 보장)

* **Amazon ElastiCache (Redis)**
  * **인프라 역할:** 세션 토큰 및 선착순 대기열 처리를 전담하는 인메모리 스토어로서 메인 데이터베이스의 I/O 병목을 방지합니다.
  * **서비스 작동:** 타임딜 마감 직전 초 단위로 갱신되는 선착순 잔여 수량과 실시간 대기열 번호표를 메모리 상에서 빛의 속도로 연산하여 표기합니다.

* **Amazon RDS (MySQL)**
  * **인프라 역할:** 유저 정보, 주문, 결제 데이터의 트랜잭션 정합성과 원자성(Atomicity)을 보장하는 메인 RDBMS입니다.
  * **서비스 작동:** 선착순 경쟁이 종료된 후 최종 공구에 성공한 주민들의 회원 데이터, 주문서, 결제 영수증을 유실 없이 안전하게 복구 가능한 상태로 영구 저장합니다.

<hr>
#### D. 가시성 및 가용성 레이어 (장애 방어 및 백업)

* **Amazon CloudWatch**
  * **인프라 역할:** 애플리케이션 레이어로부터 실시간 구동 로그 및 인프라 메트릭을 수집하여 통합 가시성(Observability)을 확보합니다.
  * **서비스 작동:** 특정 공구방에 트래픽이 비정상적으로 폭주하여 에러 시스템 임계치를 넘을 경우, 즉시 로그를 적재하고 개발자 팀에게 실시간 경고 알림을 발송합니다.

* **Amazon S3**
  * **인프라 역할:** RDS의 스냅샷 및 증분 백업본을 이중화 저장하여 재해 복구(DR) 자산으로 활용하는 객체 스토리지입니다.
  * **서비스 작동:** 인프라 전체에 예기치 못한 재해가 발생하더라도 결제 장부와 주민들의 데이터가 유실되지 않도록 매일 새벽 RDS의 복사본을 안전하게 격리 보관합니다.

### 구조 설계
- **가상 서비스 주제:** [우리동네 기반 타임딜 공동구매 플랫폼]
- **개요:** 특정 지명 없이 사용자의 현재 GPS 좌표를 기반으로 동네 주민들끼리 대용량 생필품을 묶음 공동구매하거나 배달비를 아끼기 위해, 마감 시간을 정해두고 선착순으로 공구 방을 개설 및 참여하는 하이퍼로컬(Hyperlocal) 서비스입니다.


### 기술별 사용 이유 작성
- **MySQL (RDBMS):** 사용자의 고유 ID, 암호화된 비밀번호, 거주지 인증 정보 등 데이터의 일관성과 정확성(ACID 원칙)이 절대적으로 보장되어야 하는 핵심 유저 마스터 데이터를 저장하기 위해 사용합니다.

- **Redis (In-Memory NoSQL):** 타임딜 마감 직전 동시다발적으로 발생하는 대규모 선착순 진입 대기열 관리, 실시간 인증 세션 토큰 저장, 인기 공구 품목 캐싱 등 디스크 I/O 없이 초고속 데이터 처리가 필요할 때 부하를 분산하기 위해 도입합니다.

- **Kotlin Data Class:** 불필요한 보일러플레이트 코드(Getter, Setter, toString 등)를 배제하고, 회원 가입 및 로그인 통신 과정에서 오가는 DTO(Data Transfer Object) 객체의 안정성과 가독성을 극대화하기 위해 활용합니다.

- **Coroutine (async/await):** 로그인 유효성 검증이나 외부 위치 API 동기화 시 발생하는 무거운 I/O 작업 동안 메인 스레드를 차단(Blocking)하지 않고 경량 비동기 스레드 스위칭을 구현하여, 선착순 트래픽 집중 환경에서도 중단 없는 고성능 서버 환경을 구축합니다.

- **GraphQL:** 사용자가 앱을 켰을 때 '메인 지도 화면(핀 좌표와 마감 시간만 필요)'과 '공구 상세 화면(참여자 명단, 댓글, 이미지 필요)'에서 요구하는 데이터 필드가 다릅니다. 기존 REST API의 고정된 엔드포인트 방식이 유발하는 오버패칭(Overfetching)을 완전히 해결하고 모바일 네트워크 트래픽을 최적화하기 위해 단일 엔드포인트 구조의 GraphQL을 필수 도입합니다.

- **JIRA:** 프로젝트의 거대한 이정표인 에픽(Epic) 아래에 실무자가 전담할 단위 업무들을 태스크(Task/Story)로 잘게 쪼개어 이슈 티켓을 직접 생성합니다. 담당자(Assignee)를 명확히 지정하여 업무 병목을 방지하고 작업 진행 상황을 칸반 보드로 시각화하여 공유합니다.

- **AWS architecture diagram:** 마이크로서비스 아키텍처(MSA) 환경에서 각 서비스가 API Gateway를 거쳐 어떻게 분산 DB와 통신하고 제어되는지 가시적인 클라우드 아키텍처 다이어그램을 도식화하여 시스템 구조의 이해를 돕습니다.

#### 개발과정 참여 방법

> 개인적으로 각자 하나씩 Micro Service를 만들어 넣으면 팀 성적도 개인 보너스 점수도 같이 올라갈 수 있다고 생각합니다. 화이팅!

#### 1. 어떤 서비스를 할 것인가
- **어떤 서비스가 존재하는가:** 기존 당근마켓의 중고거래나 일반 대형 이커머스의 전국 단위 배송 서비스가 존재합니다.

- **나는 어떤 불편함을 겪은 적 있나:** 1인 가구가 대용량 생필품(식재료, 화장지 등)을 묶음으로 저렴하게 사고 싶어도 보관 공간과 양이 너무 많아 포기하거나, 혼자 배달 음식을 시킬 때 최소 주문 금액과 고액의 배달비 부담을 겪은 적이 있습니다.

- **내가 정말 스마트폰에서 원하는걸 할 수 있나:** 스마트폰의 핵심 기능인 GPS 기반 실시간 위치 인식을 활용하면, 멀리 떨어진 사람이 아닌 '지금 나와 같은 아파트 단지나 골목에 있는 인접한 이웃들'을 매칭하여 즉석에서 1/N 타임딜 공동구매를 체결할 수 있습니다.

#### 2. 해당 서비스를 위해서 어떤 일을 해야하는가 (User-Service 관점)
- **어떤 요청이 들어오는가:** 유저의 회원가입 요청(ID, 암호화된 PW, 닉네임, GPS 기반 동네 인증 정보)과 로그인 유효성 검증 및 토큰 발급 요청이 들어옵니다.

- **어떤 정보가 같이 들어올 수 있는가:** 모바일 기기 고유의 디바이스 토큰, 현재 위도/경도 좌표 정보, 클라이언트가 요청하는 화면 단위 헤더 정보가 함께 유입됩니다.

- **어떤 정보를 찾아야하는가:** 현재 로그인 세션을 요청하는 유저가 유효한 세션 토큰을 보유하고 있는지, 거주지 인증 만료 시간이 지나지 않았는지 실시간 상태 정보를 찾아야 합니다.

- **어떤 데이터를 어디에서 찾아야하는가:** 초고속 조회가 필요한 일회성 인증 토큰 및 세션 데이터는 **Redis(In-Memory)**에서 조회하고, 영구 보존이 필요한 고유 회원 식별 정보 및 마스터 계정 데이터는 **MySQL(RDBMS)**에서 조회합니다.

- **찾은 데이터는 어떻게 가공해야하는가:** 데이터베이스에서 조회한 날것의 유저 정보를 Kotlin data class 구조의 안정적인 DTO 형태로 변환하고, 패스워드와 같은 민감 정보는 마스킹 처리하여 안전한 상태 객체로 인코딩합니다.

- **어떤 응답을 해야하는가:** 요청의 성공 여부를 나타내는 표준 HTTP 상태 코드와 함께, 인증에 성공했음을 증명하는 Access Token 및 유저 프로필 메타데이터를 담은 응답 객체를 반환합니다.

- **어떤 정보를 응답에 포함시켜야만 하는가:** 클라이언트 앱이 후속 보안 요청을 보낼 때 인증 헤더에 첨부해야 하는 'Bearer 유효 토큰'과 화면 상단에 띄워줄 유저의 '닉네임', '동네 인증 여부' 상태값은 반드시 포함해야 합니다.

- **어떤 정보를 응답에 포함시키지 말아야만 하는가:** 유저의 솔트(Salt)가 포함된 단방향 암호화 비밀번호 해시값, 데이터베이스의 Auto Increment 고유 PK(Primary Key) 번호 등 내부 보안 인프라 노출 우려가 있는 정보는 절대 응답에 포함시키지 말아야 합니다.

#### 3. API 부분을 구현하기 위해선 어떻게 해야할까?
- **HTTP API란?:** 웹상에서 컴퓨터와 컴퓨터가 데이터를 주고받기 위해, 인터넷 전송 규약인 HTTP(Hypertext Transfer Protocol) 시스템을 기반으로 클라이언트와 서버 간의 상호작용 규격을 정의한 인터페이스입니다.

- **REST API란?:** 자원(Resource)을 고유한 URI로 표현하고, 행위(Verb)를 HTTP Method(GET, POST, PUT, DELETE)로 정의하며, 표현(Representation) 형식을 JSON 등으로 주고받는 자원 지향적(Resource-Oriented) 아키텍처 스타일의 API 규격입니다.

- **Spring Boot 프레임워크에서의 API 작성:** `@RestController` 어노테이션을 클래스 상단에 선언하여 JSON 형태의 데이터를 반환하는 컨트롤러를 지정하고, 각 기능별 매핑 어노테이션(`@PostMapping`, `@GetMapping`)과 `@RequestBody`를 통해 들어오는 요청 객체를 비즈니스 서비스 로직으로 바인딩하여 처리합니다.

#### 4. TDD를 해보려고 노력한다면 어떻게 해야할까?
- **어떤 Test를 만들어야할까 (TDD 단위 테스트 시나리오):** 실제 프로덕션 코드를 치기 전에 기능의 명세를 검증하는 단위 테스트를 먼저 작성합니다.
	- *성공 케이스:* 올바른 이메일 규격과 암호화 조건이 포함된 데이터 구조가 입력되었을 때, `201 Created` 코드와 함께 정확한 회원가입 완료 객체가 반환되는지 검증.
	- *실패(예외) 케이스:* 이미 가입된 중복 아이디로 가입을 시도하거나 필수 파라미터가 누락(null)되어 유입될 때, 서버가 터지지 않고 `400 Bad Request` 예외 에러와 명확한 실패 메시지를 반환하는지 검증하는 방어 코드를 선제 구축합니다.

- **Small/Medium/Large 테스트의 분리 기준:**
	- *Small Test (단위 테스트):* 네트워크나 디스크 I/O, 데이터베이스 연결 없이 오직 메모리상에서 단일 메서드나 비즈니스 로직의 순수 로직만을 초고속(밀리초 단위)으로 검증하는 테스트입니다. (Mock 객체 적극 활용)

	- *Medium Test (통합 테스트):* 단일 서버 내부에서 여러 컴포넌트 간의 상호작용을 검증하며, 스프링 컨텍스트를 로딩하거나 데이터베이스 로컬 연결(H2 데이터베이스 등)을 포함하는 내부 통합 테스트입니다.
	
	- *Large Test (시스템/E2E 테스트):* 실제 운영 환경과 유사하게 네트워크 통신, 외부 API 연동, 클라이언트와 분산 데이터베이스 전체 인프라가 유기적으로 엔드투엔드(End-to-End)로 완벽히 작동하는지 검증하는 대규모 환경 테스트입니다.

#### 5. GraphQL
- **뭘까?:** 페이스북에서 최초로 개발한 웹 API용 데이터 쿼리 언어(Query Language)이자, 클라이언트가 서버로부터 필요한 데이터를 효율적으로 요청할 수 있도록 돕는 런타임 엔진입니다.

- **어케 쓸까?:** 서버 측에서 제공할 수 있는 데이터 타입과 필드를 스키마(Schema)로 미리 엄격하게 정의해 두고, 클라이언트는 해당 스키마를 기반으로 원하는 데이터 구조를 쿼리(Query) 또는 뮤테이션(Mutation) 문법에 담아 단일 엔드포인트(`/graphql`)로 HTTP 요청을 보내 사용합니다.

- **왜 쓸까?:** 한 번의 요청으로 연관된 여러 분산 데이터를 결합하여 가져올 수 있어 통신 횟수를 줄여주며, 서버가 고정된 데이터를 내려주는 것이 아니라 클라이언트가 화면 요구사항에 맞게 구조를 주도하여 가져올 수 있기 때문에 프론트엔드와 백엔드의 개발 의존성을 낮춰줍니다.

- **우리 프로젝트에서의 필요성 (역할):** 우리동네 타임딜 플랫폼의 메인 화면인 '동네 지도'에서는 오직 공구 방의 위치 좌표와 타이머 정보만 있으면 되지만, 유저가 특정 핀을 눌러 상세 창을 열 때는 상세 글, 참여자 프로필, 댓글 목록이 전부 필요합니다. 화면별로 API 엔드포인트를 여러 개 파거나 무거운 데이터를 다 전송할 필요 없이, 단 하나의 GraphQL 엔드포인트로 각 화면 진입 시점마다 꼭 필요한 데이터 필드만 쏙쏙 골라 서빙하여 모바일 환경의 하이퍼로컬 통신 효율을 극대화하는 중추적 역할을 수행합니다.

#### 6. 어떤 DBMS를 쓸까?
- **어떤 DBMS가 존재하는가?:** 행과 열로 데이터를 엄격하게 관리하는 관계형 데이터베이스(RDBMS)인 **MySQL, PostgreSQL** 등이 존재하며, 고정된 스키마 없이 키-값(Key-Value)이나 문서(Document) 형태로 고속 처리를 지원하는 NoSQL 데이터베이스인 **Redis, MongoDB** 등이 존재합니다.
- **어떤 역할인가? (언제 사용하는가?):** - *RDBMS (MySQL):* 회원 계정 정보, 정산 내역, 결제 기록 등 데이터 간의 명확한 관계성이 필요하고 절대 데이터 정합성이 깨지면 안 되는 강력한 트랜잭션이 요구되는 핵심 원천 데이터 보관 시 사용합니다.
	- *NoSQL (Redis):* 실시간 트래픽 폭주 시 발생하는 대기열 번호표 관리, 중복 로그인 방지용 유저 세션 확인, 자주 조회되는 동네별 핫딜 정보 캐싱 등 초고속 동시성 제어가 필요한 캐시 메모리 레이어로 활용할 때 사용합니다.

- **DBMS:** MySQL (Persistent 저장소), Redis (In-Memory 캐시 및 대기열 레이어)
