# Conversation Log - 타임딜 vs 오더 갑론을박

## 배경

팀 프로젝트 주제: **타임딜 공동구매 서비스**

- 민상 담당: 타임딜 (공동구매 참여 처리)
- 연제 담당: 주문 서비스 (Order Service)

---

## 쟁점

민상이와 연제가 각자 담당 기능을 정리하다가 겹치는 부분이 많다는 걸 발견.

- 둘 다 Redis + RDBMS + Coroutine + TDD 기술 스택 사용
- 둘 다 선착순 처리, 중복 방지 로직 포함
- 언뜻 보면 같은 기능을 두 명이 중복으로 구현한 것처럼 보임

→ **"타임딜이랑 오더가 사실 같은 거 아니야?"** 로 갑론을박 시작

---

## 결론

알고 보니 **같은 기능이 아니라 서로 다른 레이어**였음.

| 구분 | 민상 | 연제 |
|------|------|------|
| 담당 | 공동구매 참여 처리 | 주문 로직 처리 |
| 역할 | 마감시간, 지역조건, 정원 초과 확인 후 참여 확정 | 참여 확정 이후 수량 검증, 재고 차감, 주문 상태 관리 |
| 시점 | 공구방 입장 시점 | 주문 생성 ~ 취소 시점 |

민상이 → "참여 가능한지" 판단  
연제 → "주문을 어떻게 처리할지" 실행

순서가 있는 구조이지 충돌이 아님.

---

## 추가 결정 사항

프로젝트 전체 주제가 **타임딜 공동구매 서비스**인 상황에서,  
민상이의 파트 이름이 `타임딜`이면 서비스 전체를 담당하는 것처럼 보여 혼란을 줌.

→ 민상이의 담당 파트명을 **`타임딜` → `참여 처리`** 로 리네임하기로 합의.
