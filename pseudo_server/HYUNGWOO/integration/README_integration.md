# 통합 — 타임딜 공동구매 마이크로서비스 연동 문서

## 담당 주제

여섯 개의 마이크로서비스(사용자 / 딜 생성 / 상세 조회 / 참여 처리 / 주문 처리 / 알림)를 하나의 서비스로 합칠 때, 서비스 사이를 잇는 **연동(글루) 로직**과 그 TDD를 담당합니다.

핵심은 네 가지입니다. 딜의 `metric`(인원/수량/금액)에 따라 선착순 모델을 하나로 라우팅하는 것, 마감 확정이 `metric` 기준으로 누적값을 집계해 성사 여부를 판정하는 것, 사용자 서비스가 제공하는 신원(`userId`)과 동네 정보를 모든 서비스가 일관되게 사용하도록 잇는 것, 그리고 마감 결과·주문 확인 같은 이벤트를 알림 서비스로 전달해 참여자에게 발송되도록 잇는 것입니다.

## 문서 읽는 순서

1. `integration.md`
   - 통합 서비스 개요
   - 통합 계층이 책임지는 것(선착순 모델 분기, metric 기준 마감)
   - 연동 핵심 의사코드(성공 판정기, 모델 라우팅, 누적값 집계, 마감, 이벤트 수신, 진행도)
   - 통합 라이프사이클과 경계 계약

2. `integration_spring_boot_design.md`
   - 생성 이벤트 수신(`@TransactionalEventListener`)
   - metric 기준 마감 스케줄러(`@Scheduled`)
   - 주문 서비스 집계 조회, metric 전략 분리

3. `integration_small_test.md`
   - 외부 저장소 없이 의사코드 함수의 반환값 검증(성공 판정/모델 라우팅/진행도)

4. `integration_medium_test.md`
   - 서비스 간 저장소 I/O 정합 검증(이벤트→read-model, 참여 키↔상세, 주문 집계↔마감)

5. `integration_large_test.md`
   - 인원/수량/금액 딜 각각의 생성→상세→참여·주문→마감 전체 흐름, 동시성, 취소

6. `integration_spring_boot_test.md`
   - 이벤트 핸들러, 마감 스케줄러, 집계 클라이언트, 전략, 인증 보강 테스트

## 점수 보강 포인트

- 기술 적용도: 커밋 후 이벤트 수신, metric 전략, 마감 스케줄러, 서비스 간 집계 조회를 통합 흐름에 연결
- TDD: 의사코드 기반으로 small / medium / large / Spring Boot 보강 테스트 분리
- 트러블슈팅: 선착순 모델 이중화, metric 무시 마감, 진행도 단위 불일치 문제 정의 및 해결
- 문서 구성: 통합 의사코드 / 설계 보강 / 계층별 테스트 문서 분리
- 협업 과정: 여섯 서비스가 dealId · 생성 이벤트 · Redis 키 규약 · 사용자 신원(userId/동네) · 마감 결과 알림으로 어떻게 맞물리는지 명확화
